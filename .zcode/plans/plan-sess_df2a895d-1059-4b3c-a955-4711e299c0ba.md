### Sub-sprint 9A — Validation consolidation (~2-4h)

**User decision: tighten the email check to the existing `isValidEmail()` at all 4 sites.** Justification accepted: inputs lacking a TLD (e.g. `a@b`) currently enable the admin-modal submit button but get rejected by the backend anyway, so no previously-successful submission breaks; tightening fixes a real UX gap.

**Email check — 4 sites** (`includes("@")` → `isValidEmail()`):

- `components/admin/bookings/NewBookingModal.tsx:132` — `email.includes("@")` in `isValid`
- `components/booking/BookingForm.tsx:258` — `customerEmail.includes("@")` in `isValid`
- `app/(admin)/login.tsx:154` — `email.includes("@")` in login `isValid`
- `app/(admin)/login.tsx:210` — `fpEmail.includes("@")` in forgot-password `isValid`

Each site: `import { isValidEmail } from "@/utils/validation"` and replace the `includes("@")` term. The existing `isValidEmail` is already tested (12 cases in `tests/utils/validation.test.ts`).

**Password rules — verbatim duplicate → shared helper:**

- `app/(admin)/login.tsx:87-94` (`handleResetPassword`) and `components/admin/settings/SecurityCard.tsx:83-90` (`handleChangePw`) both: `newPw !== confirmPw → "Passwords do not match."` then `newPw.length < 6 → "Password must be at least 6 characters."`
- New `utils/validation.ts` export: `validatePasswordChange(newPassword: string, confirmPassword: string): { ok: true } | { ok: false; error: string }` — pure function, returns the first failure (or `{ok:true}`). Both call sites become: `const v = validatePasswordChange(newPw, confirmPw); if (!v.ok) { setErr(v.error); return; }`.
- This preserves the exact error strings and the exact failure ordering (match-check before length-check).

**Tests (9A):**

- Extend `tests/utils/validation.test.ts`: add `validatePasswordChange` coverage — match-fail, length-fail, success (3-4 cases).
- The existing `NewBookingModal`/`BookingForm`/`login`/`SecurityCard` tests assert the visible behavior (error messages, disabled states) — they pass unchanged because the strings + ordering are preserved. Run them to confirm parity.

**What does NOT change:** `EmailGuestForm.tsx` (no email check — out of scope), the `isEmail` disambiguators in `api/admin.ts`/`bookings/index.tsx`/`AdminSidebar.tsx` (different semantic intent — search-query classification, not validation).
